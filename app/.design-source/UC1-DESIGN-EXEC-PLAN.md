## Capability-spec contracts (connected system)
- SINGLE CHILD PROFILE = SOURCE OF TRUTH: every parent screen reads from and writes to one child profile (mood, milestones, stars, activities, avatar all live there). Parent and kid apps are 'two faces of one brain' with realtime sync. No parent screen may introduce a sibling data store or a verdict/score that isn't a read of this profile.
- LOOP 1 EMOTIONAL INSIGHT: kid mood check-in -> Parent Today feed -> emotional milestone (Self-Regulation domain) -> Coach context. The parent Today feed and Coach MUST surface kid-originated mood data so the parent 'understands without prying.'
- LOOP 2 GUIDED ACTION: Milestone/Growth flags a gap -> Coach/AI recommends -> Parent ASSIGNS an activity -> it becomes the kid's quest. There must be a parent-side 'assign activity' action that crosses into kid mode; Growth->Coach->Academy/Activity assignment is one chain.
- LOOP 3 MOTIVATION & HABIT: complete quest/comic -> earn stars+streak -> Treasure unlocks -> returns tomorrow. Parent surfaces (Today feed, Profile activity, streak chip) reflect kid stars/streak/quest completions.
- LOOP 4 HERO AVATAR EVERYWHERE: ONE saved on-device avatar rendered through a single shared canvas module across Hero Card, comic covers, 6 comic panels, Academy hero, Studio stamp, reward/badge art, breathing buddy, milestone celebration. Build rule: add a surface by passing a new template to the shared module, never re-implement compositing. Parent surfaces that show the child (Profile child-card, milestone celebrations, shareable Hero Card) must render via this same engine.
- LOOP 5 GROWTH STORY: quests+comics+moods+activities -> milestones move -> Journal auto-logs -> shareable Hero Card / Weekly Report. The growth story compounds into a SHAREABLE artifact; Journal auto-logging and the Weekly Sunday digest are the connective outputs.
- JOURNAL AUTO-LOG CONTRACT: the Journal auto-logs moments (from kid activity/mood/milestones) AND accepts manual notes; entries carry an AUTO (Arbor-generated) vs MANUAL provenance flag and a Development-Map DOMAIN tag tying each moment to one of the 7 clinical domains.
- DEVELOPMENT MAP AS THE SPINE: 7 clinical domains / 35+ milestones are the shared taxonomy threaded through Today (dev-map card), Growth (the Map), Behaviors (resolved events feed Self-Regulation), Journal (per-entry domain chip), Academy (courses matched to gaps), and Care (consult packet mirrors domains). Every cross-surface reference uses the SAME 7 framework.json domain labels.
- COACH IS DATA-PERSONALIZED: the AI coach is personalized by the child's data (milestones, mood, behavior logs, lens) and is reachable as a connective endpoint from every surface (deep-link with seeded context). It also runs the parent-approved memory review queue (append-only ledger).
- GUIDANCE-NOT-DIAGNOSIS / CLINICAL FIREWALL: Arbor is a developmental guidance tool, never diagnostic; for any concern it routes to a verified professional (Care Network). No 0-100 score, verdict tag, or weakest-domain deficit pointer may appear on child data on any parent screen.
- KID MODE IN THE WEB APP: the parent web app embeds a Kid Mode (web-app handoff exposes kidMode/kidAvatar/kidStars/kidQuest/exitKid tokens) — 'the full parent dashboard plus a Kid Mode, always in sync.' Mobile & web are at full parity (both worlds, 6 SEL games, hero comics).
- BILINGUAL/RTL & ON-DEVICE PRIVACY ARE SYSTEM-WIDE CONTRACTS: every screen, story, and game flips He/En as layout (not just text); avatar renders on-device and child data stays put until the parent exports (GDPR/COPPA).

## Per-screen connected-system obligations
- TODAY: the live activity feed must surface KID-ORIGINATED events (mood check-ins, quest/comic completions, stars/streak) not just parent-logged moments — this is the Loop-1 + Loop-3 parent-facing window. The 'one daily action' hero is the entry to Loop-2 guided action. (Plan currently derives the feed only from parent playLogs/behaviorLogs/milestones — must include kid mood + quest/star events from the shared profile.)
- GROWTH (Development Map): must expose the Loop-2 chain explicitly — a gap on the Map feeds Coach recommendation feeds an ASSIGNABLE activity that becomes the kid quest. The plan adds a 'feeds the Map/Academy/Care' hint banner but omits the 'assign activity -> kid quest' affordance that closes Loop-2.
- BEHAVIORS: the 'Linked to Development Map' navy card (resolved events feed Self-Regulation) is the connective hook and is correctly planned — but it must also feed Journal auto-log and Coach context, not just deep-link to Growth.
- JOURNAL: must render the AUTO vs MANUAL provenance badge AND the per-entry 7-domain chip, and the auto entries must include kid-side moments (mood, quest, milestone crossings) auto-logged from the shared profile — this is the Loop-5 'Journal auto-logs' output. (Plan maps provenance heuristically and risks treating Journal as parent-log-only.)
- ACADEMY: courses must be matched to Development-Map GAPS (Loop-2: 'expert courses, matched to gaps') and the Learning Map must share the same 7-domain taxonomy + domain->icon/tone map used by Growth. The development copilot lives here (per deck).
- CARE NETWORK: the redactable consult packet must be built from the live child profile and its section labels must mirror the 7 Development-Map domains (Growth->Care threading); it is the 'route to a verified professional' endpoint of the guidance-not-diagnosis contract.
- PROFILE: the child-card avatar MUST render through the shared avatar canvas module (Loop-4), the activity feed must show real kid+parent events from the shared profile, and Family Circle must read live ShareGrants (role-based co-parents/carers = the deck's 'Family Circle'). Profile is also where 'multiple children' + kid-activity tracker + Arbor Plus live.
- ASK ARBOR (Coach): must remain data-personalized (seeded with child profile/lens/mood context) and keep the parent-approved memory review queue (the append-only ledger that the growth story reads). It is the hub every other surface deep-links into with context.
- SHELL/IA: Ask Arbor as a first-class category is consistent with the deck's 6-category model (Today, Growth, Academy, Ask Arbor, Care Network, Profile). The Shell must also preserve the Kid Mode entry/route (KidModeButton/KidModeOverlay) because the web app explicitly contains an in-sync Kid Mode — not merely 'out of scope wiring.'
